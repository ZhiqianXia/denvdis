# denvdis 数据获取机制详解

## 核心概念：两阶段过程

```
┌─────────────────────────────────────────────────────────────┐
│                    第1阶段：NVIDIA编译时                    │
├─────────────────────────────────────────────────────────────┤
│ CUDA编译器 → 生成SASS表/CICC/PTX宏 → 嵌入到官方工具二进制  │
│ (nvdisasm, ptxas, libnvrtc-builtins.so等)                  │
│ 时机: CUDA工具包发布时期                                   │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                   第2阶段：用户事后提取                      │
├─────────────────────────────────────────────────────────────┤
│ denvdis工具 → 从二进制中提取已嵌入的数据表 → 保存到文件   │
│ (denv, denv11, denv12, cic12, deptx, nvb)                  │
│ 时机: 用户数据分析时期                                     │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                   第3阶段：用户利用数据                      │
├─────────────────────────────────────────────────────────────┤
│ 用户代码 → 使用提取的表进行优化/分析/逆向 → 编译或分析    │
│ 时机: 用户项目开发时期                                     │
└─────────────────────────────────────────────────────────────┘
```

---

## 详细说明

### 第1阶段：NVIDIA编译官方工具（编译时💾）

**发生在**: CUDA工具包发布前

```
NVIDIA内部开发流程:
┌──────────────────────────────────────┐
│ CUDA编译工具源代码                   │
│ ├─ compiler/frontend              │
│ ├─ ptx-optimizer                  │
│ ├─ sass-encoder (SASS指令编码表)   │
│ ├─ cicc-data (中间代码数据)        │
│ └─ ptx-macros (PTX宏定义)          │
└──────────────────────────────────────┘
            ↓ 编译链接
┌──────────────────────────────────────┐
│ 官方工具二进制文件                   │
│ /usr/bin/nvdisasm                 │ ← SASS表在.data节中
│ /usr/bin/ptxas                    │ ← PTX宏在.data节中
│ /lib/libnvrtc-builtins.so.XX      │ ← BC代码嵌入库中
└──────────────────────────────────────┘
```

**关键事实**:
- NVIDIA在编译官方工具时，将SASS表、CICC数据、PTX宏等**硬编码**到二进制的`.data`节中
- 这些数据已经是**加密存储**的形式（XOR-Salt-LCG加密）
- 每个CUDA版本对应不同的嵌入数据和种子

**为什么这样做**?
- 防止逆向工程（虽然不是绝对安全）
- 分发打包方便（整个工具链自包含）
- 性能优化（运行时直接访问，无需外部文件）

---

### 第2阶段：denvdis工具提取（事后提取🔓）

**发生在**: 用户安装denvdis项目后运行提取工具

```
denvdis工作流:
┌────────────────────────────────────────┐
│ 官方工具二进制 (已编译)              │
│ /usr/bin/nvdisasm                   │
└────────────┬───────────────────────────┘
             │ denv/denv11/denv12读取
             ↓
┌────────────────────────────────────────┐
│ ELF解析 (ELFIO库)                    │
│ ├─ 查找.data节                      │
│ ├─ 定位偏移和大小                   │
│ └─ 提取加密数据                     │
└────────────┬───────────────────────────┘
             │ XOR-Salt-LCG解密
             ↓
┌────────────────────────────────────────┐
│ LZ4解压 (denv11/12)                   │
│ ├─ 检查压缩标记                     │
│ ├─ 调用LZ4解压                      │
│ └─ 验证完整性                       │
└────────────┬───────────────────────────┘
             │ 逐架构处理 (sm30, sm75等)
             ↓
┌────────────────────────────────────────┐
│ 输出文件                              │
│ data11/sm75/      (SASS表)           │
│ data11/sm80/      (SASS表)           │
│ cicc12/llvm       (CICC中间代码)    │
│ macros/1          (PTX宏)            │
└────────────────────────────────────────┘
```

**关键事实**:
- denvdis**不参与编译过程**，只做事后提取
- 工具本身**不生成**这些表，只是从已有的二进制中提取
- 提取的数据格式和内容**由NVIDIA决定**，denvdis只负责解密和解压

---

### 第3阶段：用户利用数据（应用时📊）

**发生在**: 用户使用denvdis提取的数据进行其他工作

#### 场景1：SASS编码参考
```
用户流程:
提取的data11/sm75/ ← 查看SASS指令编码表
                   ← 理解指令格式
                   ← 用于逆向工程或优化分析
```

#### 场景2：内核性能分析
```
用户编译代码:
kernel.cu → nvcc编译 → kernel.cubin
            ↓
            使用官方nvdisasm反汇编
            ↓
            参考denvdis提取的SASS表
            ↓
            分析性能特征
```

#### 场景3：自定义编译工具
```
构建自定义编译器:
denvdis提取的表 ← 理解SASS编码规则
               ← 自定义代码生成
               ← 优化或补丁生成
```

## 各工具的具体数据来源

### denv, denv11, denv12 - SASS表来源

**原始来源**: NVIDIA CUDA编译器内部

```
NVIDIA编译流程:
PTX → SASS编码 → 编码表 → 嵌入nvdisasm.data节

各CUDA版本的嵌入方式:
CUDA 10.1: 原始表（未压缩）→ data/目录
CUDA 11.x: LZ4压缩表 → data11/目录
CUDA 12.x: LZ4压缩表（含新架构）→ data12/目录
```

**denvdis的角色**:
- ❌ 不生成SASS表
- ✓ 提取已嵌入的表
- ✓ 解密XOR加密
- ✓ 解压LZ4压缩（v11+）
- ✓ 按架构整理成易用的目录结构

---

### cic12 - CICC中间代码来源

**原始来源**: NVIDIA LLVM编译器后端

```
NVIDIA编译流程:
PTX → LLVM IR → CICC格式 → 序列化 → 嵌入nvdisasm.data节

CICC包含内容:
├─ 编译器符号表 (c4CA7760)
├─ 编译配置     (c4CA7380)
├─ LLVM IR代码  (llvm) - LZ4压缩
├─ PTX输出      (ptx_out) - LZ4压缩
└─ 寄存器分配信息 (regs) - LZ4压缩
```

**denvdis的角色**:
- ✓ 从nvdisasm二进制提取CICC块
- ✓ 解密和解压各组件
- ✓ 保存为可读格式供分析

---

### deptx - PTX宏来源

**原始来源**: ptxas汇编器的内置宏库

```
NVIDIA构建ptxas时:
PTX宏定义库 → 编译汇编器 → 嵌入ptxas.data节

PTX宏分布:
1-7:   Fermi时代 (compute_20等)
8-20:  Kepler/Maxwell时代
21+:   现代时代 (Pascal, Volta等)
```

**denvdis的角色**:
- ✓ 从ptxas二进制定位宏块
- ✓ 解密宏定义
- ✓ 保存为独立文件

---

### nvb - LLVM Bitcode来源

**原始来源**: NVIDIA运行时库的嵌入IR代码

```
NVIDIA构建libnvrtc-builtins时:
LLVM Bitcode(各架构) → 链接到库 → 导出符号

运行时库结构:
libnvrtc-builtins.so.XX
├─ getBuiltinHeader() → 编译器header
├─ getArchBuiltins(arch) → 架构相关BC
│  ├─ arch=1: Fermi BC
│  ├─ arch=20: Kepler BC
│  ├─ arch=30: Maxwell BC
│  └─ ...
└─ 其他函数
```

**denvdis的角色**:
- ✓ 动态加载库 (dlopen/dlsym)
- ✓ 穷举调用getArchBuiltins(1-128)
- ✓ 保存返回的BC数据为文件

**特殊之处**:
- 唯一一个**不涉及解密**的工具
- 利用运行时符号导出的正式接口

## 编译链中的位置

```
用户编译CUDA代码时:

kernel.cu
   ↓ nvcc (使用NVIDIA编译器内核)
   ├─→ 查询嵌入的SASS表 (来自官方工具)
   ├─→ 查询嵌入的PTX宏 (来自ptxas)
   └─→ 调用libnvrtc库 (包含BC代码)
   ↓
kernel.cubin (包含编译结果)

用户分析时:
   ↓
使用denvdis提取上述嵌入数据
   ↓
理解编译器做了什么 & 进行优化
```

**关键理解**:
- 用户编译时，NVIDIA官方工具**自动使用**嵌入的数据
- 用户通常**不需要显式使用** denvdis提取的数据（除非做特殊分析）
- denvdis是用于**学习、逆向、分析**官方工具决策的工具

## 数据时间版本关系

```
编译时版本 vs 提取程序的关系:

官方工具版本              提取工具              输出数据特征
────────────────────────────────────────────────────────
CUDA 10.1                 denv              原始未压缩
   ↓
CUDA 11.0-11.8            denv11            LZ4压缩（统一种子）
   ↓
CUDA 12.0-12.5            denv12            LZ4压缩+新架构
   ↓
CUDA 12.6+                denv12 (仍可用)   继续支持（可能需更新种子）
```

**重要**:
- denvdis的提取工具版本需**配套**官方CUDA版本
- 错配版本会导致：
  - ❌ 解密失败（种子不匹配）
  - ❌ 数据损坏（解压失败）
  - ❌ 文件格式错误

## 实际数据流示例

### 场景：分析CUDA 12.x的SASS编码

```
编译时 (NVIDIA):
────────────────
CUDA 12.2的编译工具链
├─ nvcc编译器
├─ nvdisasm反汇编器 (包含SASS表)
└─ ptxas汇编器 (包含PTX宏)
   ↓ 编译时嵌入加密SASS表
/usr/bin/nvdisasm (二进制, ~80MB)
   │ .data节中: 加密SASS表 (sm50-sm120)
   └─ 加密方式: XOR-Salt-LCG
      压缩方式: LZ4

用户事后提取:
───────────
$ ./denv12 /usr/bin/nvdisasm
   ↓
   1. 打开nvdisasm ELF文件
   2. 查找.data节
   3. 定位各架构的SASS表偏移
   4. 读取加密数据块
   5. XOR-Salt-LCG解密
   6. LZ4解压
   7. 按架构保存
   ↓
data12/
├─ sm75/          ← Turing SASS表 (解密+解压后)
├─ sm80/          ← Ampere SASS表
├─ sm89/          ← Ada SASS表
├─ sm90/          ← Hopper SASS表
└─ sm120/         ← Blackwell SASS表

用户利用:
────────
查看data12/sm90/中的指令编码规则
  ↓
理解Hopper架构的SASS如何编码
  ↓
优化自己的Hopper内核
  或 进行SASS级逆向工程
```

## 为什么这样设计？

### NVIDIA的考虑

1. **编译器自包含**
   - 无需额外下载编码表文件
   - 分发包更完整

2. **防止篡改**
   - 加密阻止简单修改
   - 难以注入恶意代码

3. **版本绑定**
   - 编码表与编译器版本紧密关联
   - 防止版本混用

### denvdis的补充价值

1. **学习工具**
   - 了解编译器内部工作
   - 学习SASS指令编码

2. **分析工具**
   - 创建自定义优化
   - 进行性能分析

3. **安全研究**
   - 理解加密机制
   - 分析编译器行为

---

## 总结：是编译时吗？

### ✅ 是编译时
- **NVIDIA官方工具的编译时**: SASS表等数据在CUDA工具包发布前生成并嵌入
- **概念上的"编译时"**: 这些表是编译过程的一部分

### ❌ 不是用户编译时
- **用户代码编译时**: 不需要手动使用denvdis
- **自动使用**: 官方工具已包含，用户透明使用

### 🔓 denvdis的角色
- **事后提取+解密**: 从已发布的官方工具中反向提取这些编译时数据
- **学习和分析**: 供用户理解编译器决策的工具

---

**示意图总结**:

```
NVIDIA编译器开发        CUDA工具包发布        用户使用
────────────────────────────────────────────────────
生成SASS表             打包成nvdisasm         用户运行nvcc
   ↓                        ↓                    ↓
嵌入+加密              官方工具二进制          自动使用表
   ↓                   (包含加密表)            (透明)
→ denvdis提取点 ←
   (事后工作)         事后才能反向提取
                      (需要解密+解压)
```

答案: **编译时生成 + 事后提取**
